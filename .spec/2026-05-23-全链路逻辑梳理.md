# MySkills 全链路逻辑梳理

> 2026-05-23
> 涵盖：Processor 架构、数据流、输入/输出契约、素材-口播匹配逻辑、Remotion 渲染链路

---

## 1. 系统总览

```
输入                    Pipeline             输出
GitHub URL ────── github-promo ──────┐
                                     ├── final.mp4（字幕+混音）
                                     ├── timeline.json / video_config.json / .srt
                                     └── 中间产物（各 Processor 产出）
```

### 1.1 唯一 Pipeline

| Pipeline | 入口 | Processor 链 | 对应目录 |
|----------|------|-------------|----------|
| `github-promo` | GitHub repo URL | RepoAnalyzer → ScriptTimelineComposer → MediaGenerator → VideoRenderer → PostProducer | `repo-analyzer/` → `script-timeline-composer/` → `media-generator/` → `video-renderer/` → `post-producer/` |

### 1.2 Processor 架构原则

- 每个 Processor = **输入契约 + 处理逻辑 + 输出契约**
- 不关心上下游是谁，只保证自己输入输出符合契约
- Pipeline DAG 由 `pipelines/github-promo.json` 定义边（edges），Orchestrator 拓扑排序后顺序执行
- 支持**断点续跑**：checkpoint 文件 `$OUTPUT_DIR/.pipeline_checkpoints.json` 记录已完成步骤

### 1.3 目录命名约定

| 类型 | 规则 | 示例 |
|------|------|------|
| Processor 目录 | kebab-case | `repo-analyzer/`, `media-generator/`, `post-producer/` |
| Python 包目录 | 下划线（符合 PEP8） | `media_generator/`（即 `media-generator/media_generator/`） |
| Processor 名称 | PascalCase | `RepoAnalyzer`, `ScriptTimelineComposer` |
| Pipeline 文件 | kebab-case | `github-promo.json` |
| 输出目录时间戳 | `YYYY-MM-DD-HHMM` | `2026-05-23-1452` |

## 2. Processor 0: RepoAnalyzer（数据接入 + 深度分析 + 内容创作）

### 2.1 角色 + 目录
- **角色**：AI agent 主导，**一个上下文完成全部分析 + 创作**。输入 GitHub URL，输出结构化内容 + 已下载素材。
- **合并自**：原 ContentIngester + InsightAnalyzer + MaterialCollector
- **口播创作时机**：全部分析完成后、综合所有上下文信息创作
- **目录**：`repo-analyzer/`

### 2.2 数据流（4 Phase）

```
GitHub repo URL
    │
    ├── Phase 1: Repo 探索（只读不存） ────────────────────────────
    │   ├─ gh api repos/{owner}/{repo}        → 元数据（stars, topics, description）
    │   ├─ gh api repos/{owner}/{repo}/readme → README 全文（base64 decode）
    │   ├─ gh api repos/{owner}/{repo}/contents → 文件树扫描
    │   ├─ gh api repos/{owner}/{repo}/git/trees → 完整目录结构
    │   ├─ 产品/官网/Demo 页面浏览（可选）→ 了解项目展示形态
    │   └─ Agent 实时消化，不写磁盘缓存
    │
    ├── Phase 2: 素材发现 + 主动创作 ────────────────────────────
    │   ├─ README 素材发现（架构图、benchmark、demo GIF、代码段）
    │   ├─ 按 4 维评分 1-5（详见 §2.8）
    │   ├─ 评分 >= 3 的素材自动下载到 materials/
    │   ├─ 素材不足时的主动创作（详见 §2.9）：
    │   │   ├─ 源码分析后生成架构图 / 数据流图（mmdc/dot 渲染）
    │   │   ├─ Playwright 对核心页面截图（README、GitHub 首页、官网等）
    │   │   ├─ Playwright 录制滚动演示视频（scroll_video）
    │   │   └─ 上述创作素材自动注册到 material_manifest.json，评分 5
    │   └─ material_manifest.json（素材清单）
    │
    ├── Phase 3: 源码深度分析（按需读取，不存磁盘） ───────────────
    │   ├─ 基于文件树 + README 判断项目类型，选择最有价值的文件
    │   ├─ gh api repos/{owner}/{repo}/contents/{path} 实时读取
    │   ├─ 按照项目类型做针对性分析（详见 §2.4）
    │   ├─ 架构图 / 数据流图生成（详见 §2.5）
    │   └─ 分析结果直接写入 content.json，不缓存源码文件
    │
    └── Phase 4: 内容创作（综合全部上下文） ─────────────────────
        ├─ 口播脚本 8-15 段：综合以下全部上下文信息创作
        │   ├─ 仓库元数据（项目名、描述、stars、topics）
        │   ├─ README 理解（项目定位、功能亮点、使用方式）
        │   ├─ 已下载素材视觉内容（logo、demo GIF、截图、代码段）
        │   ├─ 源码分析洞见（架构亮点、性能卖点、核心创新点）
        │   └─ 架构图/数据流图（图表内容的自然语言讲解）
        ├─ **素材绑定：每段口播直接关联最能展示该段内容的素材**
        │   └─ script.segments[].{primary_material, material_refs}
        ├─ 封面提示词（3:4 + 16:9，中英文各一份）
        ├─ 发布文案（标题、正文 100-200 字、标签 6-8 个）
        └─ content.json（含 source_code_insight.dimensions + 素材绑定）
```

### 2.3 输入/输出

| | 路径 |
|--|------|
| **输入** | GitHub repo URL |
| **输出** | `content.json` (ContentModel) + `material_manifest.json` (MaterialManifest) + `materials/` 目录 |

### 2.4 源码分析维度（项目类型适配）

**分析方式随项目类型变化**，不套用固定维度模板。Agent 先识别项目类型，再针对性分析。

#### 2.4.1 项目类型识别

Agent 根据仓库描述、README、文件结构判断项目类型：

| 项目类型 | 特征 | 示例 |
|----------|------|------|
| **AI/ML 模型** | 有 model/、train/、inference/ 目录，含 .pt/.bin/.onnx 权重文件或引用，含 transformer/diffusion/LLM 等术语 | bytedance/Lance, meta/llama, stability-ai/stable-diffusion |
| **开发者工具** | CLI 入口（main.rs/main.go/main.py）、SDK/API 封装、包管理器配置 | vitejs/vite, curl/curl, vercel/hyper |
| **框架/库** | 有核心 lib/src 目录，API 接口定义，包发布配置 | expressjs/express, django/django, tensorflow/tensorflow |
| **应用/产品** | 有 UI 层、前端代码、Dockerfile、部署配置 | nextcloud/server, mattermost/mattermost |
| **基础设施** | 网络/存储/容器相关，有 deploy/、k8s/、infra/ 目录 | kubernetes/kubernetes, nginx/nginx |
| **算法/学术** | 论文实现、实验脚本、Jupyter notebook、数据集 | 各种 paper 实现仓库 |

#### 2.4.2 AI/ML 模型项目分析（重点）

对于 AI 模型类项目，Agent 以**顶尖 AI 专家视角**分析以下维度：

| # | 维度 | 分析内容 | 视频用途 |
|---|------|----------|----------|
| 1 | **模型架构总览** | 整体网络结构（Transformer/DiT/SSM/扩散/混合专家等），参数规模，层数，核心模块组成 | 架构图场景的口播素材 |
| 2 | **输入/输出设计** | 输入张量形状、模态（文本/图像/视频/多模态）、tokenizer/vae 编码方式、输出解码器设计 | 展示模型能力范围 |
| 3 | **核心创新点** | 相比 prior art 的改进：新的 attention 机制、loss 设计、训练策略、架构简化等 | 视频核心卖点 |
| 4 | **数学直觉** | **从线性代数/几何直观/信息论等视角解释网络设计动机**：为什么用某个激活函数、attention 的几何意义、扩散过程的概率解释、位置编码的频域原理等 | 技术深度讲解的亮点 |
| 5 | **训练管线** | 训练数据规模与来源、loss 函数设计、优化器、学习率策略、分布式训练方案（FSDP/DeepSpeed 等） | 训练门槛说明 |
| 6 | **推理/部署** | 推理引擎、量化方案（FP16/INT8）、vLLM/TensorRT 等加速、内存优化、批处理策略 | 实际可用性佐证 |
| 7 | **代码级 "Aha!" Moment** | 3-5 行最精妙的实现片段（如 flash attention 的 tiling 技巧、高效的 kernel fusion） | 视频中直接展示的代码特写 |

分析结果输出到 `content.json → source_code_insight.dimensions`。

#### 2.4.3 其他项目类型分析

非 AI 类项目保持通用分析框架，Agent 根据项目类型灵活调整重点：

| # | 维度 | 说明 |
|---|------|------|
| 1 | **架构模式** | 整体架构风格（微服务/插件化/P2P/事件驱动）、模块划分、核心设计模式 |
| 2 | **核心数据流** | 从输入到输出的完整链路，关键管道/通道，异步/同步模型 |
| 3 | **模块依赖关系** | 核心模块间依赖方向、耦合程度、依赖图中的关键路径 |
| 4 | **性能与扩展性** | 并发模型、缓存策略、批处理、水平扩展能力 |
| 5 | **技术栈深度** | 语言版本、关键依赖、编译/构建方式、跨语言调用 |
| 6 | **工程成熟度** | CI/CD、测试策略、错误处理、文档完善度 |
| 7 | **代码亮点** | 最精妙的代码片段、最巧妙的优化、最特别的实现手法 |

**跨语言通用**：以上分析完全由 AI agent 通读源码后判断，不依赖任何语言特定的静态分析工具。agent 能理解 Python、Rust、Go、TypeScript、Java、C++ 等任何语言。

### 2.5 架构图 / 数据流图生成（AI 驱动 + 通用渲染）

**核心思路**：Agent 阅读源码后理解架构 → 用 AI 能力组织成结构化描述 → 通过通用渲染工具出图。完全不依赖语言特定的调用图工具。

```
Agent 读完代码后
    │
    ├─ 理解项目类型，选择需要生成的图类型
    │
    ├─ 通用项目图类型：
    │   ├── 架构总览图（Architecture Overview）
    │   │   展示：模块分层 + 模块间通信方式（HTTP/gRPC/消息队列/shared memory）
    │   │   用途：开场场景展示项目整体架构
    │   │
    │   ├── 核心数据流图（Data Flow Diagram）
    │   │   展示：输入 → 处理管道 → 输出的完整链路，标注关键步骤
    │   │   用途：展示"从用户请求到返回结果"的路径
    │   │
    │   └── 模块依赖关系图（Module Dependency Graph）
    │      展示：核心模块 + 依赖方向 + 接口边界
    │      用途：展示解耦程度和架构清晰度
    │
    ├─ AI/ML 模型项目额外图类型：
    │   ├── 模型架构图（Model Architecture Diagram）
    │   │   展示：各层/模块的连接关系，标注张量形状（B×L×D）、注意力头数、
    │   │   FFN 隐藏层维度、是否 causal mask 等关键参数
    │   │   用途：直观展示网络结构和参数规模
    │   │   示例:
    │   │   ```mermaid
    │   │   graph LR
    │   │     A[Input: B×L×D] --> B[Embedding]
    │   │     B --> C[Transformer ×N]
    │   │     C --> D{Mask?}
    │   │     D -->|Causal| E[Masked Attn]
    │   │     D -->|Full| F[Bi Attn]
    │   │     E --> G[FFN: 4D→D]
    │   │     F --> G
    │   │     G --> H[Output: B×L×D]
    │   │   ```
    │   │
    │   ├── 训练/推理管线图（Training Pipeline）
    │   │   展示：数据加载 → 前向 → loss → 反向 → optimizer 的完整流程，
    │   │   标注 FSDP/DeepSpeed 策略、混合精度位置
    │   │   用途：展示训练规模和分布式策略
    │   │
    │   └── 张量流图（Tensor Flow Diagram）
    │      展示：核心算子的张量形状变化过程
    │      用途：展示前向/反向传播的数据变换逻辑
    │
    ├─ Agent 用 Mermaid 语法写图描述：
    │   ```mermaid
    │   graph TD
    │     A[用户输入] --> B[CLI Parser]
    │     B --> C{CUDA 可用?}
    │     C -->|是| D[GPU Pipeline]
    │     C -->|否| E[CPU Fallback]
    │     D --> F[模型推理]  --> G[后处理] --> H[输出]
    │     E --> F
    │   ```
    │
    │   或 Graphviz DOT:
    │   ```dot
    │   digraph { rankdir=LR;
    │     Input -> Parser -> Inference -> Output;
    │     Inference -> {GPU, CPU}[style=dashed];
    │   }
    │   ```
    │
    └─ 使用通用渲染工具将图描述转为图片：
       ├─ `mmdc -i arch.mmd -o materials/arch_overview.png`  (Mermaid CLI)
       ├─ `dot -Tpng arch.dot -o materials/data_flow.png`    (Graphviz)
       └─ 素材注册到 material_manifest.json，评分 5
```

使用的渲染工具（**仅做图渲染，不做源码分析**）：

| 工具 | 用途 | 安装 | 跨语言？ |
|------|------|------|----------|
| Mermaid CLI (`mmdc`) | 架构图、数据流图、序列图等 | `npm install -g @mermaid-js/mermaid-cli` | 不限语言 |
| Graphviz (`dot`) | 有向/无向依赖图、层级图 | `apt install graphviz` | 不限语言 |

### 2.6 ContentModel 结构

```python
ContentModel:
  source: GitHubSourceMeta          # full_name, url, stars, forks, topics, license
  content: NormalizedContent        # title, tagline, points[3-5], summary, domains
  script: Script                    # full_text, segments[{text, duration_est,
                                    #   primary_material, material_refs}], total_duration_est
  covers: Covers                    # 3x4 + 16x9, 中英文各一份
  publish_copy: PublishCopy         # titles, body (100-200 字), tags[6-8]
  source_code_insight: ...          # 7 维源码分析（架构模式、数据流、模块依赖、性能...）
  meta: Meta                        # generated_at, source
```

### 2.7 MaterialManifest 结构

```python
MaterialManifest:
  version: "2"
  repo: RepoRef | None
  created_at: str | None
  materials: list[Material]      # id, type, source, path, score, metadata
```

素材类型（枚举 `contracts/enums/materials.json`）：
- `logo` — 项目 logo
- `demo_gif` — 演示 GIF
- `image` — 图片（架构图、benchmark、概览图等）
- `screenshot` — UI / 页面截图
- `code_snippet` — 代码段
- `scroll_video` — GitHub/官网页面滚动录屏
- `link_video` — 外部嵌入式视频链接
- `source_code` — 有分析价值的源码文件
- 其他：`social_proof`, `repo_stats`, `changelog`, `comparison`, `doc_page` 等

### 2.8 素材数量约束（铁律）

RepoAnalyzer 必须遵守以下素材数量约束，确保下游 ScriptTimelineComposer 有足够但不过量的素材进行语义匹配：

**发现阶段（material_discovery.json）**

| 约束 | 下限 | 上限 | 说明 |
|------|------|------|------|
| 素材总数（discovered） | 14 | 35 | 从 README 中发现的全部候选素材 |
| 最终下载数（downloaded） | 12 | 25 | 评分 >= 3 的素材 |
| 素材类型覆盖数 | 4 种 | — | 至少覆盖 4 种不同类型（见下） |

**按类型数量约束（下载后）**

| 素材类型 | 最少 | 最多 | 说明 |
|----------|------|------|------|
| `logo` | 1 | 1 | 项目 logo 或图标，必须有且仅有一个 |
| `demo_gif` / `link_video` | 3 | 8 | Demo 演示 GIF / 视频，优先覆盖不同功能点 |
| `image` / `screenshot` | 2 | 6 | 架构图、概览图、Benchmark、UI 截图等 |
| `code_snippet` | 1 | 4 | 核心代码段（安装、快速开始、API 调用） |
| `scroll_video` | 0 | 3 | GitHub/官网页面滚动录屏 |
| `source_code` | 0 | 5 | 有分析价值的核心源码文件 |
| `social_proof` / `repo_stats` / 其他 | 0 | 3 | 社会证明、仓库统计等 |

**素材主动创作**：若 README 发现的素材不足以达到数量下限，Agent 应主动创作（详见 §2.9 素材主动创作）。

**质量门槛**

| 条件 | 处理方式 |
|------|----------|
| 评分 < 3 | 丢弃不下载 |
| 评分 >= 3 且 < 4 | 下载，标记为次要素材 |
| 评分 >= 4 | 下载，标记为主要素材 |
| 分辨率 < 200×200 | 自动丢弃（除非是 `code_snippet` 或 `source_code`）|
| 重复内容 / shields.io 等黑名单域名 | 自动丢弃 |

**数量违规处理**

| 情况 | 处理 |
|------|------|
| 发现的 logo > 1 | 只保留评分最高的 1 个 |
| demo_gif / link_video > 8 | 按评分降序保留前 8 个，优先保留不同功能点的演示 |
| 总素材 < 12 | 触发素材主动创作（§2.9），仍不足 12 则报错退出 |
| 类型覆盖 < 4 种 | 触发素材主动创作补充缺失类型 |

**代码段评分细化**（替代通用 4 维评分）：

| 所属章节 | 分数 | 说明 |
|----------|------|------|
| Quickstart / Installation / Getting Started | 5 | 最核心的代码段 |
| API / Example / Usage / Configuration | 4 | 重要但不关键 |
| Contribution / License | 0 | 视频中无意义 |
| 无语言标记且 > 5000 字符 | 自动 0 | 太大且无法语法高亮 |
| 无语言标记且 > 200 行 | 自动 0 | 同上 |

### 2.9 素材主动创作

当 README 发现的素材不足以满足数量下限（§2.8）时，Agent 应主动创作素材补充。按优先级：

| 优先级 | 创作方式 | 工具 | 产出类型 | 说明 |
|--------|----------|------|----------|------|
| 1 | **架构图 / 数据流图生成** | Agent 理解源码 → Mermaid/DOT → `mmdc`/`dot` 渲染 | `image` | 从源码理解架构后生成，评分 5（详见 §2.5） |
| 2 | **核心页面截图** | Playwright `page.screenshot()` | `screenshot` | README 渲染页、GitHub 首页、产品官网、Demo 页面等核心位置全屏截图 |
| 3 | **Demo 页面滚动录屏** | Playwright 脚本录制 | `scroll_video` | GitHub/官网/Demo 页面的滚动操作视频，三阶段速度曲线，~250px/s |
| 4 | **代码段提取** | Agent 从源码中摘取精华片段 | `code_snippet` | 安装命令、API 调用示例、核心算法片段等 |

**截图与录屏的 Playwright 执行规范**（`playwright/recorder.mjs`）：
- 以 1920×1080 打开页面
- 三阶段速度曲线滚动（0-8% 加速 → 8-90% 匀速 → 90-100% 减速）
- 目标速度 ~250px/s，最多占视频总时长的 50%
- scroll_video 输出为 .webm 后 ffmpeg 转 .mp4
- 截图直接输出 PNG

所有主动创作的素材自动注册到 `material_manifest.json`：
- 架构图/数据流图 → type `image`，评分 5
- 页面截图 → type `screenshot`，评分 4
- 滚动录屏 → type `scroll_video`，评分 4
- 代码段 → type `code_snippet`，评分 5

## 3. Processor 1: ScriptTimelineComposer（场景编排）

### 3.1 角色
**AI Agent 主导**。素材已由 RepoAnalyzer 绑定到每段口播，Agent **不再做素材匹配**。直接基于 content.json 的 script.segments[]（含 primary_material）逐一决策每个场景的**全部视觉维度**，输出 timeline.json + video_config.json。

> `timeline_composer.py` 仅作为工具库被 Agent 按需调用，提供 BGM 曲线、字幕拆分等确定性计算（详见 §3.4）。

### 3.2 Agent 编排工作流（4 步推理 + 3 步工具调用）

```
content.json（script.segments[] 已含 primary_material）
    │
    ├── Step 1: Agent 逐段理解口播 + 素材 ────────────────────────
    │   读取 script.segments[]，每段已有：
    │   ├─ text + duration_est     ← 口播内容
    │   ├─ primary_material        ← RepoAnalyzer 已绑定的最佳素材 ID
    │   └─ material_refs           ← 相关素材 ID 列表
    │   Agent 理解：这段口播讲什么 + 用什么素材展示
    │
    ├── Step 2: Agent 决策场景全部维度（核心） ───────────────────
    │   对每段逐一决策以下所有维度（详见 §3.3 全表）：
    │   ├─ seg_type        → 场景类型
    │   ├─ layoutId        → 布局
    │   ├─ motionMap       → 元素动效映射
    │   ├─ bgType          → 背景
    │   ├─ wrapperType     → 媒体容器
    │   ├─ transitionIn    → 入场过渡
    │   ├─ transitionOut   → 出场过渡
    │   ├─ content         → 内容字段调整
    │   ├─ staggerOrder    → 入场错峰顺序
    │   ├─ styleId         → 样式主题
    │   └─ durationSeconds → 场景时长
    │
    ├── Step 3: Agent 编排音频 ──────────────────────────────────────
    │   按 seg_type 分配 BGM 音量和 SFX（规则同前）：
    │   ├─ hook      → bgm_volume=0.3, fade_in 0.5s, whoosh SFX
    │   ├─ cta       → bgm_volume=0.3, fade_out 2s
    │   ├─ code      → bgm_volume=0.2, keypress SFX
    │   ├─ features  → bgm_volume=0.25, pop SFX
    │   └─ showcase  → bgm_volume=0.25
    │
    ├── Step 4: Agent 写入 timeline.json + video_config.json ────────
    │   - timeline.json v2：含 segments（含全部决策）、chapters、audio
    │   - video_config.json：供 Remotion 渲染消费
    │   - 校验：每个 segment 的 primary_material 存在于 material_manifest.json
    │
    ├── Step 5: [工具] 调用 BGM 曲线生成 ─────────────────────────────
    │   uv run python timeline-composer/scripts/timeline_composer.py bgm-curve \\
    │       timeline.json --output timeline.bgm_curve.json
    │   曲线写入 video_config.json → audio.bgm.volumeCurve
    │
    ├── Step 6: [工具] 生成字幕 ──────────────────────────────────────
    │   uv run python timeline-composer/scripts/timeline_composer.py subtitles \\
    │       timeline.json --output timeline.srt
    │
    └── Step 7: [工具] 划分章节 ──────────────────────────────────────
         uv run python timeline-composer/scripts/timeline_composer.py chapters \\
             timeline.json --output timeline.json（覆盖写入 chapters）
```

### 3.3 场景决策维度全表

对每段 scene，Agent 必须逐一决策以下全部维度：

| # | 决策维度 | 可选值来源 | 决策依据 |
|---|---------|----------|----------|
| 1 | **seg_type** | Agent 从 `contracts/enums/layouts.json` 读取 seg_type 枚举 | 首段→hook，末段→cta；中间段由口播内容+素材类型决定：code_snippet→code_showcase，demo_gif→showcase，benchmark→proof，其余→feature |
| 2 | **layoutId** | Agent 枚举 `video-renderer/remotion/src/layouts/index.ts` 注册的布局组件 + `contracts/enums/layouts.json` 默认映射 | seg_type + 素材类型 + 内容长度：logo+标题→hero-center，长文本+图片→split-left-text，多媒体→media-gallery，代码→code-carousel/code-display，纯文本→kinetic-typography |
| 3 | **motionMap** | Agent 枚举 Zod MotionType 联合类型（`VideoConfig.schema.ts`）+ `contracts/enums/motions.json` 角色→动效默认映射 | 元素角色+期望节奏：标题→arc-entrance/spring-elastic，副标题→scale-fade/smooth-scale-up，正文→spring-slide-up，要点→staggered-grow |
| 4 | **bgType** | Agent 枚举 Zod BgType 联合类型 | seg_type + 节奏对比：hook→nebula-3d/starfield，code→geometric，showcase→fluid-gradient 保证前后段背景不相邻重复 |
| 5 | **wrapperType** | Agent 枚举 `video-renderer/remotion/src/wrappers/` 下所有容器组件 | 素材类型+布局：代码→device-frame(macbook)，截图→glow，普通图片→perspective-frame，纯文本→无 |
| 6 | **transitionIn** | Agent 读取 Zod TransitionType + `contracts/enums/transitions.json` 过渡类型 | 与上一场景的关系：内容连续→crossfade，话题转换→slide-in/whip-pan，节奏加快→快切 |
| 7 | **transitionOut** | 同上 | 与下一场景的关系（同上） |
| 8 | **content** | 从 content.json 的 segment 字段提取，按布局需求映射到 `{title?, subtitle?, headline?, body?, points?, visual?, code?, language?}` | hero-center→title+subtitle，split-left-text→headline+body+visual，media-gallery→headline+points |
| 9 | **staggerOrder** | `["title", "subtitle", "body"]` 等元素角色顺序数组 | 视觉节奏：标题先入→主体后入，或素材先入→文字后入。默认 3-5 帧间隔 |
| 10 | **styleId** | Agent 读取 `contracts/enums/styles.json` 中所有 styleId + 视觉效果描述 | 项目整体风格+seg_type：科技感→neon-blue/tech-cyan，高端→gold-luxury/dark-purple。整视频保持 1-3 种主题 |
| 11 | **durationSeconds** | 1-300 秒，通常 = voiceover 时长 × 1.15~1.3 | 口播时长 + 15-30% 余量，让素材有展示时间。首末段可多 +1-2s |
| 12 | **bgm_volume** | 0-1（推荐：hook→0.3, 其余→0.2~0.25） | seg_type 默认 + 有口播时闪避 |
| 13 | **sfx** | Agent 读取 `contracts/enums/sfx.json` 中 seg_type→sfx 默认映射 | seg_type：hook→whoosh, code→keypress, feature→pop, cta→无 |

**全局约束**：
- seg[0] 必须是 `hook`，seg[-1] 必须是 `cta`
- 相邻场景的 bgType 不能相同（避免视觉疲劳）
- transitionIn 的第 1 个场景固定为 `crossfade`（durationFrames=1）
- 每个素材最多被一个 segment 引用为 primary_material
- 总 scene 数 = content.json 的 script.segments 长度，不合并不拆分

### 3.4 BGM 曲线生成（确定性工具）

`_generate_bgm_curve()` 生成 `{time, volume}[]` 曲线，写入：
- `timeline.bgm_curve.json`（独立文件，供 audio_mixer.py 消费）
- `video_config.json → audio.bgm.volumeCurve`（供 VideoComposer 消费）

规则：
| 场景类型 | BGM 行为 |
|----------|----------|
| hook | 从 0 淡入到 0.15~0.3（1.5s），有口播则闪避到 min(vol, 0.15) |
| cta | 最后 1.5s 从 0.3 淡出到 0 |
| 有口播段 | 闪避到 min(seg.bgm_volume, 0.15) |
| 无口播段 | 保持 seg.bgm_volume（默认 0.5） |

### 3.5 输出

Agent 完成 Step 1-5（语义推理）后直接写出 timeline.json + video_config.json。Step 6-8（工具调用）补全 BGM 曲线、字幕、章节。

| 输出文件 | 格式 | 消费方 |
|----------|------|--------|
| `timeline.json` | TimelineModel | PostProducer (audio_mixer.py) + 验证 |
| `timeline.srt` | SRT 字幕 | PostProducer (burn_subtitles) |
| `timeline.bgm_curve.json` | `[{time, volume}]` | PostProducer (audio_mixer.py --bgm-curve) |
| `video_config.json` | Zod VideoConfig | VideoRenderer (Remotion) |
| `video_config.json → audio.bgm.volumeCurve` | `[{time, volume}]` | VideoComposer.tsx BgmWithCurve |

### 3.6 全局参数计算

```
total_seconds = max(60, min(300, ceil(total_duration_est * 1.2 + 30)))
```

最少 60s，最多 300s。公式 = 口播时长 × 1.2 + 30s 余量。

## 5. Processor 3: VideoRenderer（Remotion 渲染）

### 5.1 角色
机械执行 `npx remotion render`。输入 `video_config.json`，输出 `video.mp4`。

### 5.2 渲染流程（VideoComposer.tsx）

```
video_config.json (Zod VideoConfig)
    │
    ├─ Zod 运行时校验 validateVideoConfig()
    │   → 校验失败则渲染全屏红色错误提示（布局/动效/过渡等枚举锁定）
    │
    ├─ getStructure(config.structureId)
    │   "timeline-adaptive" → sceneConfigs 键动态派生场景列表
    │   其他（funnel/timeline 等）→ structure.scenes[] 固定场景列表
    │
    ├─ 查找 styleTemplates → style 对象
    │
    ├─ 构建 SceneRenderData[]
    │   timeline-adaptive: Object.entries(sceneConfigs) → {id, type, durationFrames, config}
    │   layoutTypeToSceneType() 映射：hero-center→hook, split-left-text→solution, 等
    │
    ├─ buildTransitionPresentation(transitionOut, transitionIn)
    │   crossfade → fade() presentation
    │   slide-in  → slide({direction: from-left/right})
    │   whip-pan  → whipPan({direction}) 自定义正弦曲线速度
    │   null      → 硬切（不给 Transition 元素）
    │
    ├─ 计算调整后帧位置
    │   每个场景原始 durationFrames - 过渡重叠帧数
    │
    ├─ 构建 TransitionSeries 子树
    │   <TransitionSeries>
    │     <Sequence> <SceneBase> ... </SceneBase> </Sequence>
    │     <Transition presentation={fade/slide/whipPan} />
    │     <Sequence> <SceneBase> ... </SceneBase> </Sequence>
    │     ...
    │   </TransitionSeries>
    │
    ├─ BgmWithCurve（优先 config.audio.bgm.volumeCurve，回退 bgmCurve.ts runtime）
    ├─ Audio（voiceover，连续整段播放）
    ├─ ChapterProgressBar（章节进度条 overlay）
    │
    └─ Remotion 逐帧渲染 → video.mp4 (h264, CRF 18, 1080p@30fps)
```

### 5.3 SceneBase 四层架构

每个场景 = 4 层 AbsoluteFill 叠加：

```
Layer 0: 视频背景（可选）  — style.backgroundVideoUrl，循环覆盖程序化背景
Layer 1: 动态背景          — BackgroundLayer（starfield/bokeh/nebula-3d/aurora 等 7 种）
Layer 2: 半透明遮罩        — 线性渐变 overlay，20 帧内淡入
Layer 3: 内容布局          — LayoutDispatcher 根据 layoutId 派发到 17 种布局之一
```

### 5.4 布局系统

| `split-right-text` | SplitLeftText `direction="right"` | 同上 |
| `media-full` | MediaFull | `layouts/MediaFull.tsx` |
| `media-gallery` | MediaGallery | `layouts/MediaGallery.tsx` |
| `code-display` | CodeDisplay | `layouts/CodeDisplay.tsx` |
| `code-carousel` | CodeCarousel | `layouts/CodeCarousel.tsx` |
| `card-grid` | CardGrid | `layouts/CardGrid.tsx` |
| `stat-highlight` | StatHighlight | `layouts/StatHighlight.tsx` |
| `center-focus-video` | CenterFocusVideo | `layouts/CenterFocusVideo.tsx` |

### 5.5 layout → sceneType 映射（timeline-adaptive 模式）

定义在 `VideoComposer.tsx → layoutTypeToSceneType()`：

| layoutId | → | sceneType |
|----------|---|-----------|
| hero-center, kinetic-typography, full-screen-text | → | hook |
| split-left-text, split-right-text | → | solution |
| card-grid, floating-grid, code-carousel | → | feature |
| stat-highlight | → | proof |
| 其余（media-full, media-gallery, code-display 等） | → | showcase |

### 5.6 动画系统

3 阶段生命周期（`useElementLifecycle` hook）：
- **Entrance**（10 种）：spring-slide-up, arc-entrance, scale-fade, typewriter, reveal-mask, bounce-in, blur-focus, spring-elastic, smooth-scale-up, staggered-grow
- **Idle**（2 种）：subtle-float, glow-pulse
- **Exit**（4 种）：fade-out, slide-out-left, scale-down-out, blur-out

元素角色 → 默认动效（`motions.ts → defaultMotionMap`）：

```
title → arc-entrance    subtitle → scale-fade    headline → scale-fade
points → spring-slide-up   body → spring-slide-up   stats → scale-fade
```

每个动效关联一个 SFX（`audio/sfxLibrary.ts`）— 入口时自动触发。

### 5.7 过渡系统

使用 `@remotion/transitions` 的 `<TransitionSeries>`：

| 过渡类型 | Remotion 实现 | 视觉效果 |
|----------|--------------|----------|
| crossfade | `fade({ shouldFadeOutExitingScene: true })` | 淡入淡出 |
| slide-in | `slide({ direction: from-left/right })` | 新场景从方向滑入 |
| slide-out | `slide()` + 反向方向 | 旧场景滑出 |
| whip-pan | 自定义 `whipPanPresentation` | 方向模糊转场（正弦曲线速度 + 12px blur） |

### 5.8 VisualConfig → SceneConfig 完整字段

```typescript
interface SceneConfig {
  layoutId: LayoutType;        // 17 种枚举

  };

  transitionOut?: { type, direction?, durationFrames };
}

## 6. Processor 4: PostProducer（后期合成）

### 6.1 角色
机械执行脚本。输入 video.mp4 + 音频 + timeline，输出 final.mp4。

### 6.2 两步流水线

```
Step 1: verify_output.py
  检查所有必需文件是否存在：video.mp4, voiceover.mp3, bgm.mp3, timeline.json, timeline.srt

Step 2: audio_mixer.py
  ├─ Voiceover: loudnorm 标准化到 -16 LUFS, pad 到视频全长
  ├─ BGM: 优先 --bgm-curve 预计算曲线 → _curve_to_ffmpeg_expr()
  │      回退 segment-based 音量包络
  ├─ asplit: voiceover 分为两路 [vo_mix, vo_sidechain]
  ├─ Sidechain ducking: BGM 经 sidechaincompress（threshold=0.005, ratio=2, attack=10ms, release=100ms）
  │  → voiceover 激活时 BGM 自动闪避 ~8dB
  ├─ amix: BGM(ducked) + voiceover → mixed
  ├─ SFX: 按 timeline.audio.sfx[] 时间点 adelay + 叠层混音
  └─ ffmpeg mux: video(h264 copy) + mixed audio(aac 192k) → final.mp4

Step 2b: burn_subtitles (optional --srt)
  ├─ 查找支持 libass 的 ffmpeg
  ├─ subtitles filter: ffmpeg -vf "subtitles=timeline.srt:force_style='MarginV=24'"
  └─ 输出 final_subtitled.mp4
```

### 6.3 输出

| 文件 | 说明 |
|------|------|
| `final.mp4` | 混音后的视频（voiceover + BGM + 侧链闪避 + SFX） |
| `final_subtitled.mp4` | 额外烧录 SRT 字幕的版本 |

## 9. 跨语言数据契约

Python（Pydantic）和 TypeScript（Zod）共享同一套数据结构定义。

### 双端共享受限枚举（`contracts/enums/*.json`）

| JSON 文件 | 用途 | 同步目标 |
|-----------|------|----------|
| `layouts.json` | 15 种布局 ID + `scene_type_default_layout` 映射 | `src/enums/layouts.json` |
| `motions.json` | 18 种动效 ID + 元素角色→动效默认映射 | TypeScript 枚举 |
| `transitions.json` | 5 种过渡类型 | TypeScript 枚举 |
| `styles.json` | 12 种样式主题 ID | TypeScript 枚举 |
| `structures.json` | 5 种结构模板 ID | TypeScript 枚举 |
| `materials.json` | 15 种素材类型 + 9 种来源 + 5 种采集方式 | — |
| `sfx.json` | 11 种动效→音效文件映射 | TypeScript 枚举 |

同步方式：`video-renderer/remotion/scripts/sync-enums.sh`

### 双端模型对应

| Python (Pydantic) | TypeScript (Zod) | 序列化文件 |
|-------------------|------------------|-----------|
| `ContentModel` | — | `content.json` |
| `MaterialManifest` | — | `material_manifest.json` |
| `TimelineModel` | — | `timeline.json` |
| — | `VideoConfig` (Zod schema) | `video_config.json` |

### 接口边界

```

```
