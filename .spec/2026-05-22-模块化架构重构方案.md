## 1. 当前架构的问题

### 1.1 核心矛盾：阶段编号固定了流程

```
当前管线假设的流程是唯一的：
  GitHub URL → 分析项目 → 采集素材 → 编排时间线 → 渲染 → 合成 → final.mp4

阶段号 Phase 0~4 把这个流程写死到了目录结构、技能文档和心智模型中。
```

这导致：
- 无法轻松支持其他输入（传入一个视频文件怎么做？从零写文案怎么做？）
- 新增一个阶段就要插入 Phase 2.x，编号越来越混乱
- 每个 skill.md 都默认"我的上游是 Phase X，下游是 Phase Y"

### 1.2 重复与耦合（来自前期分析）

| 问题 | 严重程度 |
|------|---------|
| 10 组重复的 Pydantic 模型（timeline_composer vs post-producer） | 高 |
| 6 组重复的枚举常量（布局/动效/过渡等散落 5+ 文件） | 高 |
| 5 组重复的映射表（scene_type→layout 多处硬编码） | 中高 |
| 2 处跨层依赖违反（Layer 1 验证 Layer 3 的 schema） | 中 |
| 3 组重复的实用工具（BGM 曲线/ffprobe/代理配置） | 中低 |

数据契约模型的定义分散在各层，靠"手工同步"维护一致性，没有统一引用点。

## 2. 目标：处理器架构

### 2.1 核心理念

每个功能单元是一个 **处理器（Processor）**，它有：

```
┌─────────────────────────────────────────────────┐
│                 Processor                        │
│                                                   │
│  输入 Schema ──→ [ 处理逻辑 ] ──→ 输出 Schema     │
│  (契约引用)        (黑盒)        (契约引用)        │
│                                                   │
│  配置 (Config): 控制行为的参数                      │
│  元信息 (Meta): name, description, version         │
└─────────────────────────────────────────────────┘
```

**关键特性**：
- **输入/输出由契约定义**：Processor 不关心上游是谁、下游是谁，只关心收到的数据是否符合自己的输入契约
- **可插拔**：只要满足契约，任何 Processor 都可以替换另一个
- **无阶段号**：Processor 没有 Phase 1/2/3 的归属，只有功能归属
- **Pipeline 是编排**：一个 Pipeline 是一个有向图（通常是 DAG），定义 Processor 的执行顺序和数据流转

### 2.2 与当前架构的映射

当前代码按功能自然划分，已经很接近 Processor 模式。问题在于阶段编号把它们锁死在一个固定序列里。

```
当前目录               →   Processor 名称         →  功能
───────────────────────────────────────────────────────────────
content-generator       →   RepoAnalyzer             GitHub 仓库分析
material-collector      →   MaterialCurator          素材精选采集
timeline-composer      →   ScriptTimelineComposer   脚本+时间线编排
media_generation       →   MediaGenerator           音频/图片生成
video-renderer/remotion →  VideoRenderer             Remotion 渲染
post-producer          →   PostProducer             后期混音+字幕
content-source-*       →   TrendingSource            内容源（可选上游）
pipeline-orchestrator  →   Orchestrator              编排+执行引擎
```

**注意**：目录名可以不更名（避免 git history 断裂），但概念上它们变成 Processor。

### 2.3 架构总览

```
                        ┌──────────────────┐
                        │   Pipeline 定义   │
                        │  (描述文件/DSL)    │
                        └────────┬─────────┘
                                 │ 加载
                        ┌────────▼─────────┐
                        │  Orchestrator     │
                        │  编排引擎          │
                        └──┬────┬────┬─────┘
                           │    │    │
              ┌────────────┘    │    └────────────┐
              ▼                 ▼                  ▼
        ┌──────────┐     ┌──────────┐      ┌──────────┐
        │ProcessorA│────▶│ProcessorB│─────▶│ProcessorC│
        │ 分析      │     │ 创作      │      │ 渲染      │
        └──────────┘     └──────────┘      └──────────┘
              │                 │                  │
              ▼                 ▼                  ▼
        ┌───────────────────────────────────────────┐
        │               Contracts                    │
        │    (所有 Processor 之间的数据契约)           │
        └───────────────────────────────────────────┘
```

## 3. 契约层：Contracts

### 3.1 命名

```
contracts/
```

"Contract" 的语义：**模块之间的协议**。Processor 承诺产出符合 Output Contract 的数据，下游 Processor 承诺按 Input Contract 消费。不叫 Schema（太数据库）、不叫 Model（太 ORM）、不叫 Types（太编程语言）。**Contract** 准确传达了"双方协议"的含义，且在任何领域（视频、音频、文本、数据）都通用。

### 3.2 目录结构

```
contracts/                        ← 独立包，所有 Processor 的依赖
├── py/                           ← Python 端（Pydantic v2）
│   ├── __init__.py
│   ├── content.py                ← 内容分析产出
│   │   ContentModel, ContentInfo, RepoInfo, Script, ...
│   ├── material.py               ← 素材采集产出
│   │   MaterialManifest, Material, MaterialSource, ...
│   ├── timeline.py               ← 时间线编排产出
│   │   TimelineModel, TimelineSegment, ChapterMarker, SubtitleEntry, ...
│   ├── audio.py                  ← 音频相关
│   │   VoiceoverSegment, VoiceoverSplit, SfxEntry, AudioConfig, ...
│   ├── video_config.py           ← 渲染配置产出
│   │   VideoConfig, SceneConfig, TransitionConfig, ...
│   ├── pipeline.py               ← 管线编排相关
│   │   PipelineDef, ProcessorNode, Edge, ...
│   └── utils.py                  ← 通用工具（ffprobe、JSON 加载等）
├── ts/                           ← TypeScript 端（Zod）
│   ├── package.json
│   ├── tsconfig.json
│   ├── index.ts
│   ├── content.ts
│   ├── material.ts
│   ├── timeline.ts
│   └── video-config.ts
├── enums/                        ← JSON 枚举（Python + TS 共同引用）
│   ├── layouts.json              ← 15 个布局 ID + scene_type→layout 默认映射
│   ├── motions.json              ← 18 种动效 + role→motion 默认映射
│   ├── transitions.json          ← 5 种过渡类型
│   ├── styles.json               ← 12 个样式模板 ID
│   ├── structures.json           ← 4 个结构模板 ID
│   └── materials.json            ← 素材类型枚举
├── pyproject.toml
└── README.md
```

### 3.3 Python 包安装

```toml
# contracts/pyproject.toml
[project]
name = "pipeline-contracts"
version = "0.1.0"
description = "Data contracts for pipeline processor communication"
requires-python = ">=3.11"
dependencies = ["pydantic>=2.0"]
```

```bash
cd contracts && pip install -e .
```

各 Processor 在自身 `pyproject.toml` 中添加：`dependencies = ["pipeline-contracts"]`

### 3.4 注意事项

- **Python Pydantic 模型** = 运行时验证 + 序列化 + 类型提示
- **TypeScript Zod schema** = 运行时验证（Remotion 渲染入口使用）
- **JSON 枚举** = 跨语言共享的枚举值 + 默认映射表，纯数据无逻辑
- **不定义新字段**：现有所有 JSON 结构完全不变
- **`enums/layouts.json` 包含 `scene_type_default_layout`**：这只是一个"建议映射"，Processor 可以自由选择非默认值

## 4. 注册中心：Registry

### 4.1 概念

所有 Processor 向一个注册中心登记，描述自己：
- 做什么（name, description）
- 输入契约（input_contract: 引用 contracts 中的类型）
- 输出契约（output_contract: 引用 contracts 中的类型）
- 配置参数（config_schema: 可选 JSON Schema）
- 如何执行（exec: shell 命令 / Python 函数 / Docker 镜像）

### 4.2 实现方式

**轻量方案**（推荐起步）：每个 Processor 在其目录下放一个 `processor.json`：

```json
// content-generator/processor.json
{
  "name": "RepoAnalyzer",
  "description": "分析 GitHub 仓库，提取项目理解、技术栈、源码洞察",
  "input": {
    "contract": "pipeline-contracts/py/content#RepoInfo",
    "description": "GitHub 仓库信息"
  },
  "output": {
    "contract": "pipeline-contracts/py/content#ContentModel",
    "description": "完整项目分析（不含 script/covers/publish_copy）"
  },
  "config": {
    "properties": {
      "repo_url": { "type": "string", "description": "GitHub 仓库 URL" },
      "max_files": { "type": "integer", "default": 15 }
    }
  },
  "exec": {
    "type": "skill",
    "skill": "content-generator"
  }
}
```

```json
// material-collector/processor.json
{
  "name": "MaterialCurator",
  "description": "基于项目理解，目标化精选采集视觉素材",
  "input": {
    "contract": "pipeline-contracts/py/content#ContentModel",
    "description": "内容分析产出（不含 script）"
  },
  "output": {
    "contract": "pipeline-contracts/py/material#MaterialManifest",
    "description": "精选素材清单"
  },
  "exec": {
    "type": "skill",
    "skill": "material-collector"
  }
}
```

```json
// timeline-composer/processor.json
{
  "name": "ScriptTimelineComposer",
  "description": "写口播脚本 + 语义素材匹配 + 编排时间线",
  "input": {
    "contract": "pipeline-contracts/py/content#ContentModel",
    "description": "内容分析产出"
  },
  "input_extras": [
    {
      "contract": "pipeline-contracts/py/material#MaterialManifest",
      "description": "可用素材清单"
    }
  ],
  "output": {
    "contract": "pipeline-contracts/py/timeline#TimelineModel",
    "description": "时间线 + 字幕"
  },
  "output_extras": [
    {
      "contract": "pipeline-contracts/py/content#ContentModel",
      "description": "完整 content.json（含 script/covers/publish_copy）"
    },
    {
      "contract": "pipeline-contracts/py/video_config#VideoConfig",
      "description": "渲染配置（布局/动效/过渡等）"
    }
  ],
  "exec": {
    "type": "skill",
    "skill": "timeline-composer"
  }
}
```

## 5. 处理器：Processors

### 5.1 当前 Processor 清单

以下是当前项目中已有的 Processor，去掉阶段编号后的重新定义：

| Processor | 输入契约 | 输出契约 | 当前目录 |
|-----------|---------|---------|---------|
| **RepoAnalyzer** | GitHub URL（原始输入） | ContentModel（无 script） | `content-generator/` |
| **MaterialCurator** | ContentModel | MaterialManifest | `material-collector/` |
| **ScriptTimelineComposer** | ContentModel + MaterialManifest | TimelineModel + ContentModel(完整) + VideoConfig | `timeline-composer/` |
| **MediaGenerator** | ContentModel（script 部分） | 音频文件（voiceover.mp3 + bgm.mp3） | `media_generation/` |
| **VideoRenderer** | VideoConfig + timeline + materials | video.mp4 | `video-renderer/remotion/` |
| **PostProducer** | video.mp4 + voiceover + bgm + timeline | final.mp4 | `post-producer/` |
| **TrendingSource** | 无（定时触发） | repo URL | `content-source-github-trending/` |

### 5.2 Processor 不关心上下游

每个 Processor 只声明：
- "我需要什么契约"（输入）
- "我产出什么契约"（输出）
- 不声明"谁是我的上游"、"谁是我的下游"

这意味着：
- **ScriptTimelineComposer** 不需要上游一定是 RepoAnalyzer。它可以接收任何来源的 ContentModel（手动编写的、AI 生成的、从视频转录的）
- **VideoRenderer** 不需要上游一定是 ScriptTimelineComposer。它可以接收手动编写的 VideoConfig 直接渲染
- **PostProducer** 可以脱离管线独立运行，只要给它 video.mp4 + audio + timeline

### 5.3 Processor 可以是 Agent 驱动的

Processor 的 `exec` 可以是：
- **skill 调用**：AI agent 驱动的决策 + 执行（如 `content-generator` skill）
- **脚本**：确定性自动化（如 `audio_mixer.py`）
- **混合**：agent 决策 + 脚本执行（如 pipeline-orchestrator 当前的模式）

每种方式都是 Processor 的合法实现。调用者通过 `exec.type` 知道如何触发。

## 6. 管线模板：Pipelines

### 6.1 概念

Pipeline 是一个**描述性文件**，定义哪些 Processor 以什么顺序、什么数据流链接在一起。

### 6.2 Pipeline 定义格式

```json
// pipelines/github-promo.json
{
  "name": "github-promo",
  "description": "从 GitHub 仓库到推广视频的全流程",
  "version": "1.0",
  "processors": [
    { "id": "analyze",      "type": "RepoAnalyzer" },
    { "id": "curate",       "type": "MaterialCurator" },
    { "id": "compose",      "type": "ScriptTimelineComposer" },
    { "id": "audio",        "type": "MediaGenerator" },
    { "id": "render",       "type": "VideoRenderer" },
    { "id": "post",         "type": "PostProducer" }
  ],
  "edges": [
    { "from": "analyze", "to": "curate" },
    { "from": "analyze", "to": "compose" },
    { "from": "curate",  "to": "compose" },
    { "from": "compose", "to": "audio",   "output": "script" },
    { "from": "compose", "to": "render" },
    { "from": "audio",   "to": "post" },
    { "from": "render",  "to": "post" },
    { "from": "compose", "to": "post",    "output": "timeline" }
  ],
  "inputs": {
    "repo_url": { "type": "string", "description": "GitHub 仓库 URL" }
  },
  "outputs": {
    "final.mp4": { "from": "post", "description": "最终视频文件" }
  }
}
```

### 6.4 Orchestrator 职责

Orchestrator 根据 Pipeline 定义：
1. 读取 `processors` 列表，从 Registry 中加载每个 Processor 的 `processor.json`
2. 按 `edges` 确定执行顺序（拓扑排序）
3. 验证数据流中每对 Processor 的输入/输出契约兼容
4. 按顺序执行每个 Processor（可并行化无依赖的节点）
5. 在每步产出后检查 checkpoint
6. 汇报执行结果

**Orchestrator 不假设任何特定流程**。今天的 `github-promo` 用 RepoAnalyzer → MaterialCurator → ...，明天的 `video-remix` 用 VideoAnalyzer → ScriptTimelineComposer → ...，Orchestrator 不变。

### 6.5 Pipeline 的 UI 表达（远期展望）

Pipeline 定义文件也可以由图形化界面生成——用户拖拽 Processor 方块、连线、设置输入即可定义新管线。`processor.json` 中的 `input/output contracts` 确保只有兼容的 Processor 才能连接。

## 7. 重构步骤

### 7.1 Step 1：创建 `contracts/` 包

建立统一契约层，消除数据模型重复。

| 子步骤 | 内容 | 涉及文件 |
|--------|------|---------|
| 1.1 | 创建目录结构 + pyproject.toml | `contracts/py/`, `contracts/ts/`, `contracts/enums/` |
| 1.2 | 迁移 ContentModel | `content-generator/schema/models.py` → `contracts/py/content.py` |
| 1.3 | 迁移 MaterialManifest | `material-collector/schema/models.py` → `contracts/py/material.py` |
| 1.4 | 统一 Timeline 模型 | `timeline_composer.py` 内联 + `post-producer/schema/models.py` → `contracts/py/timeline.py` |
| 1.5 | 创建音频模型 | 从 post-producer/timeline_composer 提取 → `contracts/py/audio.py` |
| 1.6 | 创建 VideoConfig Pydantic | 从 schema_validator.py 迁移 → `contracts/py/video_config.py` |
| 1.7 | 创建 enum JSON 文件 | 所有枚举从各层提取 → `contracts/enums/*.json` |
| 1.8 | 创建 utils | ffprobe、JSON 工具 → `contracts/py/utils.py` |
| 1.9 | 安装验证 | `pip install -e contracts/` + 各层试 import |

**原文件保留**为 import 重定向，不破坏任何现有 import：

```python
# content-generator/schema/models.py
from pipeline_contracts import (ContentModel, ContentInfo, ...)
__all__ = ["ContentModel", "ContentInfo", ...]
```

### 7.2 Step 2：添加 `processor.json`

每个 Processor 目录添加声明文件，定义契约接口。

| 目录 | processor.json 名称 |
|------|-------------------|
| `content-generator/` | `RepoAnalyzer` |
| `material-collector/` | `MaterialCurator` |
| `timeline-composer/` | `ScriptTimelineComposer` |
| `media_generation/` | `MediaGenerator` |
| `video-renderer/` | `VideoRenderer` |
| `post-producer/` | `PostProducer` |

### 7.3 Step 3：更新各层 import

| 层 | 变更 |
|------|------|
| `content-generator/schema/models.py` | 改为 import contracts |
| `material-collector/schema/models.py` | 改为 import contracts |
| `timeline-composer/scripts/timeline_composer.py` | 删内联 Pydantic 模型，用 contracts |

## 8. 迁移策略

### 8.1 向后兼容

- **JSON 格式不变**：所有产出文件的 JSON 结构与重构前完全一致
- **旧 import 不破坏**：原文件保留为 import 重定向，现有客户代码不需要修改
- **旧管线定义仍然可用**：`github-promo` pipeline 的拓扑结构与当前 Phase 0→1→2→2.5→3→4 一致，只是换了命名

### 8.2 分步切换

```
Week 1:  创建 contracts/ 包，迁移所有模型
Week 2:  更新各层 import，删除重复
Week 3:  清理工具 + enums
Week 4:  processor.json + registry + pipeline 定义
Week 5:  测试 + 修复
```

每个步骤可以独立上线，互不阻塞。

### 8.3 风险

| 风险 | 缓解 |
|------|------|
| contracts 包版本漂移 | `pip install -e .` 开发模式，无版本号问题 |

| VideoAnalyzer 还没实现 | 目前只有 GitHub repo 管线，视频管线标记为 future work |

### 9.3 设计原则复核

| 原则 | 如何实现 |
|------|---------|
| **单一职责** | 每个 Processor 只做一件事，processor.json 和 skill.md 共同定义其职责边界 |
| **高内聚** | Processor 的输入/输出/处理逻辑/配置在同一个目录下，对外只暴露契约接口 |
| **低耦合** | Processor 之间只通过 Contracts 通信，不知道彼此的存在 |
| **无冗余** | 每个模型/枚举/映射表只在 contracts/ 中定义一次 |
| **可插拔** | 通过 processor.json 声明契约接口，Pipeline 定义决定组合方式 |
| **通用性** | 不假设特定的 Pipeline 流程，Processor 适用于任何流程 |
