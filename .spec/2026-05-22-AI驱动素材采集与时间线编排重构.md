# AI 驱动素材采集与时间线编排重构

## 问题陈述

当前视频管线在阶段划分和数据流向上有一个**根本性的顺序错误**：

### 核心问题：脚本写得太早

```
Phase 0 → 写死口播脚本 ─── Phase 1 → 盲目采集 ─── Phase 2 → 硬匹配
         ↑ 不知道有什么素材        ↑ 不读脚本          ↑ 脚本与素材脱节
```

1. **Phase 0 生成脚本时完全不知道后面会采集到什么素材**。口播说"来看这个架构图"，但 README 里根本没有架构图；口播没有预留 demo 展示的节奏点，但恰恰采集到了惊艳的 demo 视频。
2. **Phase 1 盲目全量采集时不知道脚本在说什么**。recorder.mjs 在 README 页面上全页滚动录制、全页截图，产生数百 MB 的冗余素材，但不知道哪些对口播有支撑意义。
3. **Phase 2 的机械脚本试图用关键词匹配把两段脱节的内容硬凑在一起**。基于 regex 的关键词提取和简单词频打分无法理解语义，素材匹配准确率低。

**根因**：脚本创作（需要知道有什么视觉素材可用来写配套文案）和素材采集（需要知道脚本要说什么来决定采集什么）是互相依赖的，不应该串行执行。

### 各阶段具体问题

#### Phase 1：素材采集（recorder.mjs）

1. **无差别全量采集**：打开 README 页面后全页滚动录制、全页截图、提取所有图片和视频，不考虑哪些素材真正会被使用。一个仅需 3 张截图 + 1 段 demo 视频的项目，会产生数百 MB 的冗余素材。
2. **与内容脱节**：执行时完全不读取 `repo_insights`，不知道项目核心概念是什么、技术亮点是什么、需要什么类型的视觉素材来支撑。
3. **重下载轻引用**：所有素材强制下载到本地 `materials/`，对于远程可访问的图片/视频等，引用 URL 更高效。
4. **素材质量无筛选**：不判断截图质量、不判断视频是否有意义、不判断代码片段是否对口播有支撑意义。

#### Phase 2：时间线编排（timeline_composer.py）

1. **无法做素材匹配**：脚本是 Phase 0 生成的，素材是 Phase 1 采集的，两者天然脱节。基于 regex 的关键词匹配无法理解"这个代码截图展示的就是脚本说的那个函数"。
2. **剧情流死板**：`FUNNEL_SEQUENCE` 硬编码了 hook→problem→solution→... 序列，不能根据内容叙事需求灵活编排。
3. **布局选择无意图**：seg type→layout 是固定映射表，不会考虑"这段脚本很有感染力，适合全屏文字冲击力"。
4. **过渡无意义**：所有 segment 统一 `transition_in: "crossfade"`。
5. **字幕机械截断**：固定 15 字符切分，经常在词语中间断开。

---

## 设计方案

### 核心理念

> **理解项目 → 精选素材 → 基于素材写脚本 + 编排时间线（一次完成）**

脚本创作和素材匹配是同一件事的两个面，应该在 agent 的同一个上下文窗口中完成。agent 同时看到"我有什么素材"和"我要说什么"，才能写出图文匹配的脚本。

### 架构变化

```
旧管线：
  Phase 0 (AI agent)     → content.json (含口播脚本)
  Phase 1 (机械脚本)      → material_manifest.json (盲目全量采集)
  Phase 2 (机械脚本)      → timeline.json + video_config.json (规则匹配)

新管线：
  Phase 0a: Repository Analysis (AI agent)
    → repo_insights.json (项目理解 + 源码分析 — 不含脚本)

  Phase 1: Material Curation (AI agent)
    → material_manifest.json (精选素材 + URL 引用支持)

  Phase 2: Script & Timeline Composition (AI agent — 核心合并)
    → content.json (全量 — 含口播脚本)
    → timeline.json + .srt
    → video_config.json
    → covers + publish_copy
```

**核心变化**：

1. Phase 0 拆分为 Phase 0a（项目理解）和 Phase 2 中的脚本创作部分
2. Phase 1 从机械脚本改为 AI agent 驱动的精选采集
3. Phase 2 从机械脚本改为 AI agent 语义编排，且**接过了脚本创作职责**
4. `content.json` 产出时机从 Phase 0 推迟到 Phase 2
5. 引入新中间文件 `repo_insights.json`

### 数据流全景

```
Phase 0a: Repository Analysis (AI agent)
  │  输入: GitHub repo URL
  │  输出: repo_insights.json
  │        { repo信息, 内容提炼(title/tagline/points), 源码洞察, 元信息 }
  ▼

Phase 1: Material Curation (AI agent)
  │  输入: repo_insights.json
  │  输出: material_manifest.json
  │        { 精选素材清单 + highlight_score + URL引用支持 }
  ▼

Phase 2: Script & Timeline Composition (AI agent)
  │  输入: repo_insights.json + material_manifest.json
  │  输出: content.json (含脚本)
  │        timeline.json + .srt
  │        video_config.json
  │        covers + publish_copy
  │
  ├────────────────┬────────────────┐
  ▼                ▼                ▼
Phase 2.5        Phase 3          Phase 4
(音频生成)        (Remotion)       (后期合成)
```

---

### Phase 0a: Repository Analysis（AI agent）

聚焦在**理解项目**，不写口播脚本。

#### 执行步骤

**Step 1：基础数据采集**

```bash
gh api repos/{owner}/{repo}
gh api repos/{owner}/{repo}/readme --jq '.content' | base64 -d
```
提取 `full_name, url, language, stars, forks, topics, license, ...`。

**Step 2：源码采集 + Top-15 评分选取**

与当前 content-generator 的 Step 2 相同。按权重选取 15 个最有分析价值的源码文件。

**Step 3：4 维源码分析**

与当前 content-generator 的 Step 3 相同。分析技术栈、核心逻辑、代码质量、性能安全四个维度。

**Step 4：内容提炼（不含脚本）**

生成 title / tagline / points / summary / domains / chartData / stats_text / target_users 等结构化信息。这些信息在素材采集中用于指导"需要找什么类型的视觉材料"。

**Step 5：输出 repo_insights.json**

```json
{
  "repo": {
    "full_name": "owner/repo",
    "url": "https://github.com/owner/repo",
    "language": "Rust",
    "stars": 12400,
    "forks": 380,
    "topics": ["cli", "templating"],
    "license": "MIT",
    "homepage": "https://example.com"
  },
  "content": {
    "title": "Zola",
    "tagline": "A fast static site generator in Rust",
    "points": [
      "基于 AST 的毫秒级渲染引擎",
      "原生 Sass 编译与自动前缀",
      "无运行时依赖的二进制发布"
    ],
    "summary": "重新定义了静态站点的构建体验",
    "domains": "静态站点生成、Rust 工具链",
    "chartData": [
      {"label": "Zola", "value": 95},
      {"label": "Hugo", "value": 82},
      {"label": "Next.js", "value": 70}
    ]
  },
  "source_code_insight": {
    "dimensions": { "架构": "...", "核心逻辑": "...", "代码质量": "...", "性能": "..." },
    "highlights": ["亮点1", "亮点2"],
    "analyzed_files": ["src/main.rs", "src/lib.rs", ...]
  },
  "meta": {
    "generated_at": "2026-05-22T12:00:00",
    "source": "gh-api"
  }
}
```

**schema 约束**：`repo_insights.json` 使用 `ContentModel` 的子集（不含 `script` / `covers` / `publish_copy` 字段）。虽然不强制 schema 校验，但 agent 应按此结构输出，确保 Phase 1 和 Phase 2 能可靠读取。

### Phase 1: AI 驱动的智能素材采集

Phase 1 从"运行脚本全量采集"改为"agent 读 repo_insights → 理解项目 → 策划素材需求 → 目标化采集 → 输出精选清单"。

#### 流程

```
Step 1: 读 repo_insights.json，理解项目骨架
  │  - 项目做什么的？核心概念、技术栈、领域
  │  - 技术亮点 (content.points, source_code_insight.highlights)
  │  - 什么类型的视觉素材最能支撑这个项目？
  │    （架构图？UI 截图？benchmark 图表？代码演示？demo 视频？）
  │
  ▼
Step 2: 策划素材需求清单
  │  - 列出每个需求：需要什么素材、为什么需要、获取方式、优先级
  │  - 不基于口播脚本（脚本还没写！），而基于对项目本身的理解
  │
  ▼
Step 3: 按优先级执行目标化采集
  │  - 先读页面 HTML 判断是否有价值，再截图
  │  - 严格控制录制视频时长
  │  - 支持引用 URL 而非下载
  │
  ▼
Step 4: 输出精选 material_manifest.json
```

#### Step 1：理解项目

Agent 完整阅读 `repo_insights.json`，形成：

```
这是一个 [Rust 写的静态站点生成器] 项目。
核心卖点: [毫秒级渲染、无运行时依赖]。
技术方向: [AST 转换引擎、模板编译]。
可视觉化的概念:
  → 架构图：README 中可能包含渲染管线图
  → benchmark：性能对比数据，可做成图表
  → 代码：核心算法代码块
  → CLI：命令行运行效果
```

#### Step 2：策划素材需求

**关键差异**：当前基于口播脚本来策划（脚本说啥就找啥），新方案基于项目本身来策划（项目有啥亮点就找啥）。因为脚本还没写，素材采集反而更自由——agent 可以先看看能拿到什么好素材，再决定脚本怎么写。

```
素材需求清单：

需求 1: 系统架构图 / 渲染管线图
  → 为什么：项目的核心卖点是渲染引擎，有架构图能极大增强说服力
  → 获取方式：导航 README → 找 img[alt*="architecture"] 或 img[alt*="pipeline"]
  → URL 引用可行性：如果 GitHub raw URL 可用，直接引用
  → 优先级: HIGH

需求 2: benchmark 性能数据
  → 为什么：口播一定会提到性能（核心卖点），需要数据支撑
  → 获取方式：README 中找 benchmark 表格或图表 → 截图
  → 优先级: HIGH

需求 3: CLI 使用效果截图/demo
  → 为什么：工具类项目，展示终端使用效果
  → 获取方式：README 找 CLI 示例 → 截图；或录制 10s 终端演示
  → 优先级: MEDIUM

需求 4: GitHub 项目统计数据
  → 为什么：stars/forks 数据可以展示项目影响力
  → 获取方式：gh api 已有数据，不需要采集
  → 优先级: LOW（视频中有很多方式展示数据）
```

**决策维度**：

| 维度 | 说明 |
|------|------|
| **论证强度** | 这个素材对支撑项目核心卖点的力度。架构图→强，README logo→弱 |
| **视觉独特性** | 是否有不可替代的视觉信息。benchmark 截图→高，纯文字段落截图→低 |
| **采集成本** | 获取难度。GitHub 页面截图→低，需要搭建环境录屏→高 |
| **是否可引用** | 素材是否可通过 URL 直接引用，无需下载本地 |
| **口播预留** | 这个素材未来被口播引用的可能性。架构图→高，不相关的截图→低 |

Agent 根据这些维度做取舍：
- **必须下载本地**：录制视频、经过裁剪的截图片段
- **引用 URL 即可**：GitHub raw 图片、外部可访问的视频流
- **跳过**：质量低、不相关、成本过高

#### Step 3：目标化执行采集

Agent 使用 Playwright / ffmpeg 等工具按需求清单目标化执行，不盲目全量采集：

```bash
# 1. 先读 README 的 HTML，找架构图
# 用浏览器打开 GitHub 页面
→ 滚动到包含 "architecture" 关键词的区域
→ 检查 img 标签的 alt 属性
→ 发现架构图 → 截图并保存，或记录 URL

# 2. 找 benchmark 表格
→ 在 README 中找到 benchmark 章节
→ 截图表格 → materials/benchmark.png

# 3. 有 demo 链接时才录制
→ 检查 demo URL 是否可访问
→ 仅当可用时才 ffmpeg 录制最多 15 秒
```

**采集原则**：

- 每个采集动作必须有明确的素材需求驱动，不空跑
- 先读页面文本内容判断是否有价值，再决定是否截图
- 录制视频严格控制时长（默认 ≤15s，特殊情况 ≤30s）
- 远程图片优先检查 URL 是否存活，决定下载还是引用
- 采集完成后立即质量评估

#### Step 4：输出精选 manifest

输出标准 `material_manifest.json` v2 格式，但：

- `materials[]` 中只包含精选后的素材，不是全部可能素材
- 每个素材的 `metadata.highlight_score` 表明 agent 评估的质量分数（0-1）
- 素材的 `source.url` 存原始 URL 引用
- 新增 `metadata.requirement`：该素材是为了什么需求、什么项目特征而采集

**URL 引用支持**：`path` 字段可以是远程 URL（如 `https://raw.githubusercontent.com/.../arch.png`），此时素材不占用本地磁盘空间，Remotion 渲染时通过 `staticFile()` + 网络加载。

```json
{
  "version": "2",
  "materials": [
    {
      "id": "arch-screenshot",
      "type": "screenshot",
      "path": "materials/arch-screenshot.png",
      "source": { "url": "https://github.com/owner/repo", "section": "architecture" },
      "metadata": {
        "highlight_score": 0.9,
        "requirement": "系统架构图 — 核心卖点的视觉佐证",
        "alt_text": "Parser→AST→Transformer→Codegen 渲染管线图"
      }
    },
    {
      "id": "benchmark-table",
      "type": "screenshot",
      "path": "https://raw.githubusercontent.com/owner/repo/main/docs/benchmark.png",
      "source": { "url": "https://github.com/owner/repo", "section": "benchmark" },
      "metadata": {
        "highlight_score": 0.85,
        "requirement": "性能 benchmark 数据 — 口播数据支撑"
      }
    },
    {
      "id": "demo-video",
      "type": "link_video",
      "path": "materials/demo-clip.mp4",
      "duration": 12.0,
      "metadata": {
        "highlight_score": 0.8,
        "requirement": "CLI 使用演示 — 展示实际运行效果"
      }
    }
  ]
}
```

### Phase 2: AI 驱动的语义编排（含脚本创作）

这是**新方案最核心的变化**：Phase 2 不再只是机械匹配，而是 agent 在一个上下文窗口中同时完成脚本创作 + 素材匹配 + 时间线编排。

Phase 2 同时读取 `repo_insights.json`（项目理解）和 `material_manifest.json`（有什么素材），然后**基于这两者的交集创作脚本和编排时间线**。

#### 流程

```
输入: repo_insights.json + material_manifest.json

Step 1:  理解项目 + 素材全景
Step 2:  设计叙事结构（确定场景类型序列）
Step 3:  写口播脚本（每个 segment 对应一个场景）
Step 4:  语义素材匹配（将素材分配合适的 segment）
Step 5:  布局与动效设计
Step 6:  过渡设计
Step 7:  章节划分
Step 8:  字幕生成
Step 9:  生成输出文件
         - content.json（含脚本、covers、publish_copy）
         - timeline.json + .srt
         - video_config.json
```

#### Step 1：理解全景

Agent 同时阅读两份输入，在脑中建立完整图景：

```
项目: 一个 Rust 静态站点生成器，核心卖点是毫秒级渲染
素材可用:
  ✓ 架构图 (score=0.9) — Parser→AST→Transformer 管线
  ✓ benchmark 数据 (score=0.85) — 性能对比
  ✓ CLI demo 视频 (score=0.8) — 12s 使用演示
  ✗ 没有 UI 截图（CLI 工具，无图形界面）
  ✗ 没有社区评价截图

叙事策略:
  - 开场用 benchmark 数据冲击（有数据！）
  - 中段用架构图解释"为什么这么快"（有架构图！）
  - 后段展示 CLI 使用（有 demo 视频！）
  - 由于没有 UI/social proof，这些段落在脚本中直接跳过

素材对叙事的约束和赋能:
  - 架构图的存在 → 脚本可以深入讲 AST 转换引擎
  - 无 UI 截图 → 脚本不花时间讲界面
  - benchmark 数据好 → 开场直接用数据抓住注意力
```

#### Step 2：设计叙事结构

基于项目特征和可用素材，设计叙事弧。

**不是在已有脚本上分配场景类型，而是先设计叙事结构，再按结构写脚本。**

```
项目特征:
  - 纯 CLI 工具，无 UI
  - 核心卖点是性能
  - 素材有架构图、benchmark、demo 视频

叙事结构设计:
  1. hook: 数据冲击开场（有 benchmark 素材） ← 不用常规的"介绍你是谁"
  2. solution: 介绍核心引擎（有架构图素材）
  3. showcase: CLI 使用演示（有 demo 视频）
  4. features: 功能亮点（有 points 数据，无需额外素材）
  5. cta: 总结收尾

这个结构不像 funnel（没有 problem 环节），
因为这是一个已有知名度的项目，不需要先铺垫问题。
```

**叙事结构设计原则**：

| 项目特征 | 推荐叙事策略 |
|---------|------------|
| 高知名度项目 | 直接进亮点，不需要 problem→solution 铺垫 |
| 新项目/冷门 | 问题驱动：hook → problem → solution |
| 性能亮点突出 | 数据开场：hook(数据冲击) → proof → solution |
| 代码质量是亮点 | 技术深潜：hook → code_showcase → source_highlight |
| 媒体素材丰富 | 展示主导：hook → showcase → showcase → features |
| 社区/用户驱动 | 社交证明：hook → social_proof → features → proof |

#### Step 3：写口播脚本

**关键**：此时 agent 知道每个 segment 对应什么场景类型、有什么素材可用。脚本中的每个段落都"预配对"了一个视觉呈现方案。

```
Segment 1 (hook, 5s)
  场景类型: hook
  匹配素材: benchmark-table (benchmark 截图)
  布局: stat-highlight
  脚本: "在静态站点生成器性能测试中，Zola 以 95 分位居榜首，
        比第二名 Hugo 快 15%，比 Next.js 快 35%。"
  ↑ 这个脚本片段是"为 stat-highlight 布局写的"——数据将在屏幕上放大显示

Segment 2 (solution, 8s)
  场景类型: solution
  匹配素材: arch-screenshot (架构图)
  布局: split-left-text
  脚本: "这个速度的秘密在于其基于 AST 的渲染引擎。
        Zola 将模板解析为抽象语法树，通过多阶段优化管线
        在毫秒级别完成数千个文件的渲染。"
  ↑ 这个脚本片段是"为 split-left-text + 架构图写的"——左侧文字解释，右侧展示图

Segment 3 (showcase, 12s)
  场景类型: showcase
  匹配素材: demo-video (CLI 演示)
  布局: media-full
  脚本: "使用上，Zola 保持了 Rust 工具链的一贯风格——
        一条命令初始化，一条命令构建，零配置即可上线。"
  ↑ 这个脚本片段是"为 media-full + demo 视频写的"——全屏展示 CLI 录制
```

#### Step 4：语义素材匹配

素材匹配的基础不再是关键词重叠，而是**叙事意图匹配**：

```
口播概念 "AST-based rendering engine"  ↔  架构图素材 "Parser→AST→Transformer"
  → 匹配原因：口播在解释引擎原理，架构图正好展示了这个原理的可视化表达

口播概念 "95 points benchmark"  ↔  benchmark 表格截图
  → 匹配原因：口播在报数据，表格截图让观众看到原始数据源

口播概念 "one-command setup"  ↔  CLI demo 视频
  → 匹配原因：口播在讲使用体验，视频正好展示了实际使用过程
```

**匹配准则**：

| 准则 | 说明 |
|------|------|
| 论证关系 | 素材是否能直接论证口播正在说的观点 |
| 时序关系 | 素材展示时长是否与口播段落时长匹配 |
| 互补关系 | 视觉素材是否能补充口播无法用语言表达的信息 |
| 情绪一致性 | 素材的视觉风格是否对口播的情绪基调 |

#### Step 5：布局与动效设计

Agent 为每个 segment 选择布局，基于内容 + 素材的联合推理：

```
Segment: "AST-based rendering engine"（技术严肃）
  布局推理:
    - 素材是架构图 + 口播是文字解释 → split-left-text 最自然
    - 动画 title=bounce-in（技术感），subtitle=scale-fade（平稳）
    - 左侧文字 1/3，右侧架构图 2/3 → 视觉重心在关键信息上
```

布局与动效选择表（与上一个 spec 版本相同，此处不再重复）。

#### Step 6：过渡设计

基于内容情绪而非固定默认值：

```
同主题递进 → crossfade（平滑过渡）
情绪转折 → whip-pan（强烈转向感）
章节跨越 → slide-in（空间推移感）
高潮到收尾 → crossfade（趋于平静）
```

#### Step 7：章节划分

按叙事弧分大章（多个相关 segment 聚合）：

```
Chapter 1: "Blazing Fast"             ← hook + proof 组 (benchmark)
Chapter 2: "How It Works"             ← solution 组 (架构讲解)
Chapter 3: "See It In Action"         ← showcase 组 (demo)
Chapter 4: "Get Started"              ← cta 组
```

#### Step 8：字幕生成

按语义断句而非机械按固定字符数切分。

#### Step 9：输出文件

Phase 2 一次性产出：

| 文件 | 包含内容 | 消费方 |
|------|---------|--------|
| `content.json` | 完整包含 repo + content + **script** + covers + publish_copy + source_code_insight | Phase 2.5（语音生成），最终发布 |
| `timeline.json` + `.srt` | segments + chapters + subtitles | Phase 4（字幕烧录） |
| `video_config.json` | sceneConfigs + audio config | Phase 3（Remotion 渲染） |

**特别注意**：`content.json` 现在在 Phase 2 产出（不再是 Phase 0）。schema 不变——仍为 `ContentModel` 格式。

### 整体管线流程对比

```
旧管线：
Phase 0 (Agent): 写口播脚本 + 项目分析 → content.json
Phase 1 (脚本): PLAYWRIGHT 全量采集 → material_manifest.json（数百MB冗余）
```

1. **Phase 0 从写脚本改为只做项目理解**
2. **Phase 1 从全量采集改为 AI 精选**
3. **Phase 2 从机械匹配改为 AI 编排，且接过脚本创作职责**
4. **脚本创作从素材前移到素材后**，确保图文匹配
5. **content.json 产出从 Phase 0 推迟到 Phase 2**（schema 不变）

## 验收标准

### Phase 0a (Repository Analysis)

1. 输出 `repo_insights.json` 包含足够的项目理解信息，Phase 1 能据此策划采集策略
2. 源码分析 4 维度覆盖技术栈、核心逻辑、代码质量、性能安全
3. 不包含口播脚本、封面提示词、发布文案（这些由 Phase 2 生成）

### Phase 1 (Material Curation)

1. **素材总量减少**：素材文件数减少 ≥50%，但每个关键素材都有对口播的明确支撑意义
2. **素材针对性**：每个素材能追溯回具体的项目特征需求（metadata.requirement 字段）
3. **采集效率提高**：总采集执行时间缩短 ≥30%
4. **URL 引用支持**：支持素材 path = 远程 URL，不强制全部下载本地
5. **去重**：被跳过的、无意义的素材不在 manifest 中出现
6. **向后兼容**：输出 `material_manifest.json v2` 格式完全兼容

### Phase 2 (Script & Timeline)

1. **脚本与素材共生**：脚本中的每个 segment 都有明确的素材匹配关系，而非事后硬凑
2. **叙事流合理**：segment 类型序列有清晰的叙事逻辑，不套用固定模板
3. **布局选择有推理过程**：能解释为什么选这个布局（而非默认映射表）
4. **过渡多样性**：不全是 crossfade，根据内容情绪过渡
5. **字幕完整性**：字幕按语义断句
6. **向后兼容**：`content.json` / `timeline.json v2` / `video_config.json` 格式完全兼容

### 整体

1. Phase 3/4/2.5 不需要任何修改
2. 所有数据模型（Pydantic/Zod）不变
3. `npx tsc --noEmit` 通过
4. 管线能从 repo URL 一直跑到 final.mp4

| 风险 | 等级 | 缓解措施 |
