### 评分方法：从 100 分起扣
起始 100 分。发现每项缺陷即扣分。

### 扣分规则：
1. **Technical Accuracy（技术准确性）**（扣 0-25 分）：
   - -0：所有技术论断经过验证均正确
   - -5：一处论断存在轻微不精确
   - -15：多处技术性错误
   - -25：技术描述根本性错误

2. **Narrative Pacing（叙事节奏）**（扣 0-25 分）：
   - -0：段落均衡，过渡流畅
   - -5：一个段落明显过长或过短
   - -15：多处节奏问题，流畅性被打断
   - -25：严重失衡，无法观看

3. **Audience Engagement（观众参与度）**（扣 0-25 分）：
   - -0：引人入胜的开场，全程保持兴趣
   - -5：一个段落略显枯燥
   - -15：泛泛而谈，无法吸引观众
   - -25：枯燥乏味，读起来像技术文档

4. **Structure Completeness（结构完整性）**（扣 0-25 分）：
   - -0：清晰的开场 + 包含技术架构的主体 + 结尾
   - -10：缺少开场或结尾
   - -15：没有技术架构段落
   - -25：没有可辨识的结构

### 锚定案例：
- **95 分**：技术论断准确，节奏出色，旁白引人入胜，结构完整。非常罕见。
- **75 分**：基本准确，但有一处节奏问题和中间段落略显枯燥。需要修改。
- **50 分**：多处技术错误，没有技术架构段落，语气泛泛。需要重写。
- **30 分**：技术内容根本性错误，没有结构，枯燥乏味。需要完全重写。

### 事实核查流程：
当 user prompt 中提供了源材料时，使用它们验证技术论断：

对于脚本中的每个具体技术论断：
1. **交叉验证**：将论断与 README、目录结构、依赖栈和源码进行对照
2. **分类**该论断：
   - **VERIFIED（已验证）**：源材料支持 — 不扣分
   - **UNVERIFIABLE（不可验证）**：源材料既不支持也不反驳 — 不扣分（中性）
   - **CONTRADICTED（矛盾）**：源材料明确反驳该论断 → 从 Technical Accuracy 中扣 15-25 分
   - **HALLUCINATED（虚构）**：有具体论断但毫无支持证据 → 扣 10-20 分

重要提示：证据的缺失不等于缺失的证据。只标记 CONTRADICTED（矛盾）或明显 HALLUCINATED（虚构）的论断。模糊的论断（如"现代架构"）不可验证——跳过。

### 关键规则：
- 默认状态是"找问题"。不要给予任何侥幸分。
- 当源材料可用且某技术论断被源材料 CONTRADICTED（矛盾）时，至少扣 15 分。
- 当未提供源材料时，适用原始规则：如果无法验证某个论断，至少扣 10 分。
- 一个"看起来还行"的脚本是 70-75 分，不是 85 分。
- 只有当脚本在所有维度上都真正达到生产就绪标准时，才给 85+ 分。
